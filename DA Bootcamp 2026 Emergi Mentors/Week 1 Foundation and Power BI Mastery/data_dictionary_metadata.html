Fact Tables

FactSales (Sales Transactions) FACT

Purpose Sales transaction details
Grain One row = one transaction
Row Count 100,010 transactions
Fact Type Transaction
Update Frequency Daily
Column Name Data Type Description Sample Values
SalesID Integer Unique transaction identifier 1, 2, 3...100010
Date Date Transaction date (FOREIGN KEY to DimDate) 2025-01-25, 2023-02-27
ProductID Integer Product identifier (FOREIGN KEY to DimProduct) 1166, 1101, 1159
CustomerID Integer Customer identifier (FOREIGN KEY to DimCustomer) 2248, 3962, 3927
RegionID Integer Region identifier (FOREIGN KEY to DimRegion) 5, 6, 7
ChannelID Integer Channel identifier (FOREIGN KEY to DimChannel) 1, 2, 3
Quantity Integer Units sold in transaction (MEASURE) 0, 5, 1, 2
UnitPrice Decimal Price per unit (MEASURE) 121.42, 80.15, 106.69
Discount Decimal Total discount amount (MEASURE) 7.1, 0.78, 6.15
Cost Decimal Total cost of goods sold (MEASURE) 104.92, 52.69, 53.97
Revenue Decimal Total revenue from transaction (MEASURE) 0.0, 396.85, 100.54
Critical Understanding: This is a transaction fact table. One row represents ONE sales transaction. The grain is atomic (lowest level of detail), which allows for maximum flexibility in aggregation and analysis.
Business Rules:
  • Revenue Calculation: Revenue = (Quantity × UnitPrice) - Discount
  • Gross Profit: Gross Profit = Revenue - Cost
  • Zero Quantity: Some transactions have Quantity = 0 (returns, cancellations)
  • All measures are additive: Can sum across all dimensions
Common Analysis Patterns:
  • Total Revenue by Product Category, Region, or Time Period
  • Average Transaction Value = Revenue / Transaction Count
  • Gross Margin % = (Revenue - Cost) / Revenue
  • Top Customers by Revenue
  • Channel Performance Comparison

FactMarketingSpend (Campaign Spending) FACT

Purpose Marketing campaign expenditure
Grain One row = one campaign-product-date
Row Count 30,000 records
Fact Type Transaction
Source Files 6 CSV files (appended)
Column Name Data Type Description Sample Values
Campaign Text Marketing campaign name (DEGENERATE DIMENSION) Spring Sale, EOFY Blast
Date Date Spend date (FOREIGN KEY to DimDate) 2022-02-28, 2022-02-03
ProductID Integer Product promoted (FOREIGN KEY to DimProduct) 1058, 1081, 1023
Spend Decimal Marketing spend amount (MEASURE) 1111.08, 514.35, 163.98
Marketing Campaigns (2 types across 4 years):
  • Spring Sale: Annual spring promotion (2022, 2023)
  • EOFY Blast: End of Financial Year campaign (2022, 2023, 2024, 2025)
Data Source Note: This fact table is created by appending 6 separate CSV files in Power Query. Each file contains data for one campaign in one year. Total rows = 6 files × 5,000 rows = 30,000 rows.
Marketing ROI Analysis:
  • Total Marketing Spend: Sum of Spend column
  • Marketing ROI: (Revenue / Spend) - 1
  • ROAS (Return on Ad Spend): Revenue / Spend
  • Cost per Revenue Dollar: Spend / Revenue
  • Compare campaign effectiveness by calculating ROI per campaign
  • Identify which products benefit most from marketing investment
Cross-Fact Analysis: This table connects to FactSales through the conformed DimProduct dimension. This enables questions like "Which products have high marketing spend but low sales revenue?"

FactFinancePlanning (Budget & Forecast) FACT

Purpose Monthly budget and forecast
Grain One row = one product-month
Row Count 9,600 records
Fact Type Periodic Snapshot
Update Frequency Monthly
Column Name Data Type Description Sample Values
ProductID Integer Product identifier (FOREIGN KEY to DimProduct) 1000, 1001, 1002
Month Text Month in YYYY-MM format (source data) 2022-01, 2022-02, 2022-03
MonthKey Integer Month as integer YYYYMM (FOREIGN KEY to DimMonth) 202201, 202202, 202203
MonthStartDate Date First day of month (for date relationships) 2022-01-01, 2022-02-01
BudgetUnits Integer Budgeted unit sales (MEASURE) 100, 150, 200
ForecastUnits Decimal Forecasted unit sales (MEASURE) 95.5, 148.3, 205.7
BudgetRevenue Decimal Budgeted revenue (MEASURE) 9451.72, 9108.26
ForecastRevenue Decimal Forecasted revenue (MEASURE) 8317.67, 10012.90
Critical Data Transformation: The source Excel file contains Month in text format "YYYY-MM". In Power Query, we create two additional columns:
  • MonthKey: Convert "2022-01" to integer 202201 (used for relationship to DimMonth)
  • MonthStartDate: Convert "2022-01" to date 2022-01-01 (used for date-based filtering)
Grain Mismatch Challenge: FactSales is at DAILY grain, but FactFinancePlanning is at MONTHLY grain. We cannot directly compare them without a bridge table (DimMonth) that connects monthly planning to daily sales through the date dimension.
Plan vs Actual Analysis:
  • Variance to Budget: Actual Revenue - Budget Revenue
  • Variance %: (Actual - Budget) / Budget
  • Forecast Accuracy: Compare Forecast to Actual
  • Budget YTD vs Actual YTD: Requires USERELATIONSHIP in DAX to activate DimMonth bridge
  • Track which products are over/under budget
  • Analyze forecast accuracy trends over time
Business Rules:
  • Budget set annually at beginning of year
  • Forecast updated monthly based on current trends
  • BudgetRevenue typically higher than ForecastRevenue (conservative forecasting)
  • Data covers 48 months (2022-2025) for 200 products = 9,600 rows

Business Rules & Data Quality

Data Integrity Rules:
  • All foreign keys in fact tables must have corresponding primary keys in dimension tables (referential integrity)
  • No NULL values allowed in foreign key columns
  • Date values must fall within the DimDate range (2022-01-01 to 2025-12-31)
  • ProductID values must be in range 1000-1199
  • CustomerID values must be in range 2000-3999
  • RegionID values must be 1-8
  • ChannelID values must be 1-4
Known Data Quality Notes:
  • Some product names contain minor spelling variations (e.g., "Office Supplie" instead of "Office Supplies") - this is intentional to simulate real-world data
  • Some FactSales records have Quantity = 0, representing returns or cancellations
  • Region names in dim_region.csv have inconsistent capitalization (e.g., "western australia" vs "Western Australia") - requires cleaning in Power Query
  • Customer names are anonymized as "Customer XXXX" for training purposes
Calculation Standards:
  • Revenue: (Quantity × UnitPrice) - Discount
  • Gross Profit: Revenue - Cost
  • Gross Margin %: (Gross Profit / Revenue) × 100
  • Average Transaction Value: Total Revenue / Transaction Count
  • Marketing ROI: (Revenue / Marketing Spend) - 1
  • ROAS: Revenue / Marketing Spend
  • Variance to Budget: Actual Revenue - Budget Revenue
  • Variance %: ((Actual - Budget) / Budget) × 100
Time Intelligence Requirements:
  • DimDate must be marked as a date table in Power BI
  • Date column must be selected as the date identifier
  • Continuous dates required (no gaps)
  • SAMEPERIODLASTYEAR, TOTALYTD, and other time functions will fail without proper date table configuration
  • For planning data, use USERELATIONSHIP to activate the DimMonth to DimDate bridge
Performance Recommendations:
  • Foreign keys in fact tables should be hidden in report view (enforce dimension usage)
  • Use integer keys for relationships (better performance than text keys)
  • Measures are preferred over calculated columns (calculated at query time, not stored)
  • For large models, consider incremental refresh on fact tables
  • Limit number of visuals per report page (7-10 maximum for best performance)
