Data Warehousing Fundamentals

📊What is a Data Warehouse?

A centralized repository optimized for analysis and reporting, containing historical and current data from multiple sources.
Why it matters: Separates analytics from operational systems. Enables fast queries on large datasets. Maintains data history. Provides single source of truth.
OLAP Read-Optimized Historical

⚔️OLTP vs OLAP

OLTP = Online Transaction Processing (operational). OLAP = Online Analytical Processing (data warehouse).
Aspect OLTP OLAP
Purpose Run the business Analyze the business
Operations Insert, Update, Delete Select, Aggregate
Data Age Current Historical + Current
Design Normalized (3NF) Denormalized (Star)
Users Clerks, operators Analysts, executives

📐Normalization vs Denormalization

Normalization minimizes redundancy (good for OLTP). Denormalization intentionally adds redundancy (good for OLAP).
Why denormalize for data warehousing: Fewer joins = faster queries. Business users understand flat structures better. Disk space is cheap, query time is expensive.
Normalized (OLTP): ProductCategory table: CategoryID, CategoryName Product table: ProductID, ProductName, CategoryID Denormalized (OLAP): DimProduct: ProductID, ProductName, CategoryName (CategoryName duplicated, but queries are faster)

Dimensional Modeling

⭐Kimball Methodology

Industry-standard approach to data warehouse design focusing on business processes, dimensional modeling, and user accessibility.
Core principles:
  • Focus on business processes (sales, marketing, finance)
  • Design for understandability and query performance
  • Build incrementally, deliver value early
  • Use conformed dimensions across business processes
Start with one business process (e.g., sales). Perfect it. Then expand to other processes using same dimensions.

🔢Facts vs Dimensions

Facts = Measurements (what happened). Dimensions = Context (who, what, where, when, why, how).
Characteristic Fact Tables Dimension Tables
Content Numeric measurements Descriptive text
Row Count High (millions+) Low (thousands)
Width Narrow (10-20 columns) Wide (50-100 columns)
Updates Insert only Insert + Update
Purpose Aggregate, calculate Filter, group, label

Star Schema Architecture

🌟Star Schema Pattern

Fact table in center (hub) surrounded by dimension tables (points), forming a star shape. 
           DimDate
              |
              | (1:Many)
              ↓
DimProduct → FactSales ← DimCustomer
              ↑
              | (1:Many)
              |
          DimRegion

Each dimension connects to fact with 1:Many relationship
Facts aggregate, dimensions filter
Why star schema wins:
  • Query Performance: Maximum 2 joins for any query (fact → dimension)
  • Simplicity: Business users understand it intuitively
  • Flexibility: Easy to add dimensions without remodeling
  • BI Tool Support: Optimized for tools like Power BI, Tableau
Star schema is denormalized by design. Category and Subcategory both live in DimProduct. This is intentional and correct.

❄️Snowflake Schema (Alternative)

Normalized version of star schema where dimensions have sub-dimensions. 
DimProduct → DimCategory → DimCategoryGroup
    |
    ↓
FactSales
    ↑
    |
DimCustomer → DimSegment

Dimensions are normalized (split into multiple tables)
Avoid snowflake in Power BI: More joins = slower queries. Harder for users to understand. Star schema is almost always better.

Fact Tables

🔢Fact Table Types

1. Transaction Facts
One row per business event
Example: FactSales Grain: 1 row = 1 sale transaction Columns: Date, ProductID, CustomerID, Quantity, Revenue, Cost Row count: Millions
2. Periodic Snapshot Facts
One row per time period
Example: FactInventory Grain: 1 row = 1 product on 1 day Columns: Date, ProductID, OnHandQty, OnOrderQty Row count: Products × Days
3. Accumulating Snapshot Facts
One row per process lifecycle
Example: FactOrder Grain: 1 row = 1 order lifecycle Columns: OrderDate, ShipDate, DeliverDate, OrderID Row count: One per order (updates over time)

🎯Fact Table Characteristics

Facts contain measurements (numbers) and foreign keys to dimensions
Typical structure:
  • Foreign Keys: 5-10 columns (IDs linking to dimensions)
  • Measures: 5-15 columns (numeric values to aggregate)
  • Degenerate Dimensions: Optional (transaction IDs, invoice numbers)
FactSales Structure: ├── Date (FK to DimDate) ├── ProductID (FK to DimProduct) ├── CustomerID (FK to DimCustomer) ├── RegionID (FK to DimRegion) ├── ChannelID (FK to DimChannel) ├── Quantity (Measure) ├── UnitPrice (Measure) ├── Discount (Measure) ├── Cost (Measure) └── Revenue (Measure)
Facts should be additive (can be summed across all dimensions). Revenue, Quantity, Cost are additive. Percentages and ratios are not.

Dimension Tables

📖Dimension Characteristics

Dimensions provide context (the "who, what, where, when") for analyzing facts
Key attributes:
  • Descriptive: Text, categories, hierarchies
  • Relatively Static: Changes are infrequent
  • Wide: Many columns (50-100+ attributes)
  • Denormalized: Hierarchies flattened into one table
DimProduct Structure: ├── ProductID (Primary Key) ├── ProductCode ├── ProductName ├── Category ├── Subcategory ├── Brand ├── Color ├── Size ├── Weight ├── ListPrice ├── ProductDescription └── IsActive

📅Date Dimensions (Special)

Every data warehouse needs a date dimension. Contains one row per day with calendar attributes.
Why dedicated date table:
  • Enables time intelligence (YTD, YoY, rolling averages)
  • Supports fiscal calendar (different from calendar year)
  • Includes business days, holidays, seasons
  • Required for proper time-based filtering
DimDate Structure: ├── Date (Primary Key) ├── DateKey (YYYYMMDD integer) ├── Year, Quarter, Month, Day ├── DayOfWeek, WeekOfYear ├── MonthName, QuarterName ├── FiscalYear, FiscalQuarter ├── IsWeekend, IsHoliday └── BusinessDayFlag
Generate 10+ years of dates. Include 3 years historical minimum. Mark as date table in Power BI for time intelligence functions.

🔗Conformed Dimensions

Shared dimensions used by multiple fact tables. Same structure, same keys, same values.
Why conform dimensions: Enables integrated analysis across business processes. "Which products have high marketing spend but low sales?" requires DimProduct conformed across both facts. 
        DimProduct (Conformed)
              |
      +-------+-------+
      |       |       |
      ↓       ↓       ↓
FactSales  FactMkt  FactPlanning

Same ProductID = Same Product everywhere
Enables cross-functional analysis
Build conformed dimensions once, use them everywhere. This is the key to enterprise data warehousing.

Data Grain

🎯What is Grain?

Grain defines what ONE ROW represents in a fact table. Most important modeling decision.
Why grain matters: Wrong grain = wrong aggregations. Grain must be atomic (lowest level of detail). All measures must be true at the declared grain.
✓ CORRECT Grain Definitions: FactSales: "One row = one line item on one transaction" Grain: Date × Product × Customer × Channel FactMarketingSpend: "One row = one campaign for one product on one date" Grain: Date × Product × Campaign FactFinancePlanning: "One row = one product budget for one month" Grain: Month × Product
❌ WRONG: Mixing grains in one fact table. Example: Some rows are daily, some are monthly. This breaks aggregations.
Always ask: "What business event or time period does one row represent?" Answer this before building anything.

🌉Grain Mismatches & Bridge Tables

When comparing data at different grains (daily sales vs monthly budgets), use bridge tables. 
Problem:
FactSales (daily grain) ← → FactPlanning (monthly grain)
Cannot directly compare

Solution:
        DimMonth (bridge)
            |
            ├─→ DimDate (via MonthStartDate, inactive)
            └─→ FactPlanning (via MonthKey, active)
                    
FactSales → DimDate → DimMonth → FactPlanning
Now can align daily to monthly
DimMonth Structure: ├── MonthKey (202201, 202202...) ├── MonthStartDate (2022-01-01, 2022-02-01...) ├── Year, Quarter, Month └── MonthName Relationships: - FactPlanning[MonthKey] → DimMonth[MonthKey] (Active) - DimMonth[MonthStartDate] → DimDate[Date] (Inactive) - Activated with USERELATIONSHIP in DAX

Keys & Relationships

🔐Primary Keys vs Foreign Keys

Primary Key (PK) = Unique identifier in dimension. Foreign Key (FK) = Reference to PK in fact.
DimProduct Table: ProductID (PK) | ProductName | Category 1001 | Laptop | Electronics 1002 | Mouse | Electronics FactSales Table: Date | ProductID (FK) | Revenue 2024-01-15| 1001 | 1200.00 2024-01-15| 1001 | 1200.00 ← Same product, multiple sales 2024-01-16| 1002 | 25.00 ProductID is PK in DimProduct (unique) ProductID is FK in FactSales (repeated)
One-to-Many relationship: One PK value → Many FK values. This is 95% of data warehouse relationships.

🆔Natural vs Surrogate Keys

Natural Key = Business meaning (ProductCode). Surrogate Key = System-generated (ProductID).
Aspect Natural Key Surrogate Key
Source Business system Data warehouse
Example ProductCode "P1001" ProductID 1001
Can Change Yes (rebranding) No (immutable)
Type String Integer
Performance Slower (string joins) Faster (integer joins)
Use surrogate keys (integers) for relationships. Keep natural keys as attributes for display/filtering.

🔗Relationship Cardinality

Cardinality defines how many rows on each side can relate. In Power BI: 1:Many, Many:1, 1:1, Many:Many.
✓ One-to-Many (1:*) - STANDARD DimProduct (1) → FactSales (*) One product → many sales ✓ Many-to-One (*:1) - SAME AS ABOVE FactSales (*) → DimProduct (1) Many sales → one product ✓ One-to-One (1:1) - RARE DimCustomer (1) → DimCustomerDetail (1) Usually means tables should be merged ✗ Many-to-Many (*:*) - AVOID Products (*) ↔ Promotions (*) Requires bridge/junction table
In Power BI: Many-to-many relationships are allowed but can cause performance issues and unexpected results. Use bridge tables instead.

ETL vs ELT

📥Traditional ETL

Extract → Transform → Load. Transform data BEFORE loading to warehouse.
1. EXTRACT: Read from source systems 2. TRANSFORM: Clean, calculate, join in staging area 3. LOAD: Write transformed data to warehouse Tools: SSIS, Informatica, Talend, Azure Data Factory
When to use: Complex transformations, data cleansing requirements, limited warehouse compute power, need to mask sensitive data before loading.

📤Modern ELT

Extract → Load → Transform. Load raw data first, transform AFTER in warehouse.
1. EXTRACT: Read from source systems 2. LOAD: Write raw data to warehouse (fast!) 3. TRANSFORM: Process using warehouse compute (Power Query, SQL, dbt) Tools: Power Query, dbt, Snowflake, Databricks
When to use: Cloud data warehouses (cheap compute), need to preserve raw data, iterative development, self-service analytics.
Power BI uses ELT pattern. Power Query extracts and loads, then transforms in memory. Fast and flexible.

Relationship Patterns in Power BI

➡️Filter Direction

Controls which direction filters propagate. Single = one way. Both = bidirectional. 
SINGLE DIRECTION (Recommended):
DimProduct → FactSales
Filter flows: Dimension → Fact only

User selects "Electronics" in Product filter
    ↓
DimProduct filtered to Electronics
    ↓
FactSales filtered to Electronics products only
    ↓
Report shows Electronics data

FactSales CANNOT filter DimProduct
Always use single direction unless you have a specific reason for bidirectional. Single = predictable, performant. Both = complex, slow.

❌Inactive Relationships

Relationships that exist but don't filter automatically. Activated manually in DAX with USERELATIONSHIP.
When to use inactive: Multiple date relationships (OrderDate, ShipDate, DeliverDate), grain mismatch bridges (DimMonth → DimDate), avoiding ambiguous paths.
-- Monthly planning to daily sales bridge Budget Revenue YTD = CALCULATE( [Budget Revenue], DATESYTD(DimDate[Date]), USERELATIONSHIP(DimMonth[MonthStartDate], DimDate[Date]) ) -- Activates the inactive relationship for this calculation only

⚠️Circular Dependencies

When filter paths create loops. Power BI blocks circular relationships.
Example of circular:
DimProduct → FactSales → DimCustomer → DimRegion → back to DimProduct (if bidirectional)

Solution: Keep all relationships single-direction

Slowly Changing Dimensions (SCD)

0️⃣Type 0: Fixed

Never changes. Original values retained forever.
Example: Date of Birth, Social Security Number Method: No updates allowed

1️⃣Type 1: Overwrite

Overwrite old value with new. No history preserved.
DimCustomer: CustomerID | CustomerName | Address 100 | John Smith | 123 Old St UPDATE: Address changes to "456 New Ave" CustomerID | CustomerName | Address 100 | John Smith | 456 New Ave ← Overwrote Old address lost forever
When to use: Corrections (typos), data that doesn't need history, simplicity preferred.

2️⃣Type 2: Add New Row

Insert new row with updated value. Preserve full history with effective dates.
DimProduct BEFORE: ProductID | ProductKey | ProductName | Price | EffectiveDate | ExpiryDate | IsCurrent 1 | ABC | Widget | 10.00 | 2020-01-01 | 9999-12-31 | Yes PRICE CHANGES TO $12.00 on 2024-01-01: ProductID | ProductKey | ProductName | Price | EffectiveDate | ExpiryDate | IsCurrent 1 | ABC | Widget | 10.00 | 2020-01-01 | 2023-12-31 | No 2 | ABC | Widget | 12.00 | 2024-01-01 | 9999-12-31 | Yes New row added, old row expired Facts link to correct version based on date
Type 2 is production standard. Preserves history for accurate trending and auditing. Use ProductKey for business logic, ProductID for relationships.

3️⃣Type 3: Add New Column

Add column for previous value. Limited history (1-2 prior values only).
DimCustomer: CustomerID | Name | CurrentAddress | PreviousAddress 100 | John | 456 New Ave | 123 Old St Only tracks current + 1 previous Loses history beyond that
When to use: Rarely. Only when you need exactly one prior value and Type 2 is too complex.

DAX Fundamentals

📏Measures vs Calculated Columns

Measures = Calculated at query time, context-aware. Calculated Columns = Calculated at refresh time, static.
Aspect Measures Calculated Columns
When Calculated Query time Refresh time
Responds to Filters Yes (dynamic) No (static)
Storage In memory only Stored on disk
Model Size No impact Increases size
Use For Aggregations, KPIs Row-level calculations
-- MEASURE (Preferred) Total Revenue = SUM(FactSales[Revenue]) -- Calculates based on current filter context -- CALCULATED COLUMN (Use sparingly) FullName = [FirstName] & " " & [LastName] -- Calculated once, stored, doesn't respond to filters
Default to measures. Use calculated columns only for row-level logic that needs to be used for grouping/filtering.

🎯Filter Context

Set of filters applied to data model when evaluating a measure. Created by slicers, rows/columns, visual filters.
User creates matrix: Rows: Product Category Values: Total Revenue Filter Context for "Electronics" row: - DimProduct[Category] = "Electronics" Total Revenue measure evaluates with this context: = SUM(FactSales[Revenue]) -- Only sums sales for Electronics products Different context = different result
Measures are powerful because they automatically respond to filter context. No need to write IF statements for every filter combination.

🔄Row Context

Created when iterating over table rows. Used in calculated columns and iterator functions (SUMX, FILTER).
-- Row context in calculated column Profit = FactSales[Revenue] - FactSales[Cost] -- Evaluates for EACH row -- Row context in iterator Total Profit = SUMX( FactSales, FactSales[Revenue] - FactSales[Cost] ) -- SUMX creates row context, evaluates expression per row, then sums
Important: Row context does NOT automatically filter related tables. To convert row context to filter context, use CALCULATE or RELATED.

🛡️Defensive DAX Patterns

Write DAX that handles edge cases gracefully. No errors, no blanks when they shouldn't occur.
-- ✗ BAD: Division by zero causes error Margin = [Profit] / [Revenue] -- ✓ GOOD: DIVIDE handles division by zero Margin = DIVIDE([Profit], [Revenue], 0) -- ✗ BAD: Blank when no data Customer Count = DISTINCTCOUNT(FactSales[CustomerID]) -- ✓ GOOD: Return 0 instead of blank Customer Count = VAR _Count = DISTINCTCOUNT(FactSales[CustomerID]) RETURN IF(ISBLANK(_Count), 0, _Count) -- ✗ BAD: Errors when related table empty Product Name = RELATED(DimProduct[ProductName]) -- ✓ GOOD: Handle missing relationships Product Name = IF( ISBLANK(RELATED(DimProduct[ProductName])), "Unknown", RELATED(DimProduct[ProductName]) )

Time Intelligence

⏰Time Intelligence Requirements

Built-in DAX time functions require a properly configured date table.
Requirements checklist:
  • ✓ Date table marked as date table in Power BI
  • ✓ Date column selected as the date identifier
  • ✓ Continuous dates (no gaps in calendar)
  • ✓ Unique dates (one row per day)
  • ✓ Fact table has relationship to date table
Without marking date table, functions like SAMEPERIODLASTYEAR, TOTALYTD will error or return unexpected results.

📊Prior Period Comparisons

Revenue PY = CALCULATE( [Total Revenue], SAMEPERIODLASTYEAR(DimDate[Date]) ) Revenue YoY Growth = [Total Revenue] - [Revenue PY] Revenue YoY % = DIVIDE( [Revenue YoY Growth], [Revenue PY], 0 ) Revenue PM = CALCULATE( [Total Revenue], DATEADD(DimDate[Date], -1, MONTH) ) Revenue vs Previous Month % = DIVIDE( [Total Revenue] - [Revenue PM], [Revenue PM], 0 )

📈Running Totals

Revenue YTD = TOTALYTD( [Total Revenue], DimDate[Date] ) Revenue QTD = TOTALQTD( [Total Revenue], DimDate[Date] ) Revenue MTD = TOTALMTD( [Total Revenue], DimDate[Date] ) -- Custom period (rolling 12 months) Revenue L12M = CALCULATE( [Total Revenue], DATESINPERIOD( DimDate[Date], MAX(DimDate[Date]), -12, MONTH ) )

🎯Period-to-Date Patterns

-- Year to Date Revenue YTD = CALCULATE( [Total Revenue], DATESYTD(DimDate[Date]) ) -- YTD vs Prior YTD Revenue YTD vs PYTD = VAR CurrentYTD = [Revenue YTD] VAR PriorYTD = CALCULATE( [Revenue YTD], SAMEPERIODLASTYEAR(DimDate[Date]) ) RETURN CurrentYTD - PriorYTD -- Fiscal Year (starts July 1) Revenue FYTD = TOTALYTD( [Total Revenue], DimDate[Date], "6/30" -- Fiscal year ends June 30 )

Row-Level Security (RLS)

🛡️What is RLS?

Filter data based on who is viewing it. Same report, different data per user. Enforced at dataset level.
Use cases:
  • Regional managers see only their region
  • Sales reps see only their customers
  • Multi-tenant applications (customer A can't see customer B's data)
  • Compliance requirements (GDPR, HIPAA)

📝Static RLS (Demo Only)

Hardcode values in DAX filter. Simple but not scalable.
-- Role: NSW Manager -- Table: DimRegion -- Filter: [RegionName] = "New South Wales" -- Users assigned to this role only see NSW data -- Easy to demo, but must create role per region
Static RLS doesn't scale. 50 regions = 50 roles. Use dynamic RLS in production.

🔄Dynamic RLS (Production)

Use security mapping table + USERPRINCIPALNAME() function. Scalable to thousands of users.
-- Create SecurityMap table (not related to model): UserEmail | RegionName manager.nsw@co.com | New South Wales manager.vic@co.com | Victoria admin@co.com | ALL -- Role: Regional Manager -- Table: DimRegion -- Filter: [RegionName] IN CALCULATETABLE( VALUES(SecurityMap[RegionName]), SecurityMap[UserEmail] = USERPRINCIPALNAME() ) -- How it works: -- 1. Gets current user's email (USERPRINCIPALNAME) -- 2. Looks up their regions in SecurityMap -- 3. Filters DimRegion to those regions -- 4. Filter propagates to all facts via relationships
Add new user? Just add row to SecurityMap. No changes to roles or DAX. This is production pattern.

⚙️RLS Best Practices

✓ DO:
  • Filter dimensions, not facts (better performance)
  • Use security mapping tables for dynamic RLS
  • Test roles using "View as role" before publishing
  • Document which users/groups are in which roles
  • Keep SecurityMap table unrelated to model
✗ DON'T:
  • Hardcode user emails in DAX filters
  • Create separate role for each user
  • Forget that workspace members bypass RLS
  • Use complex DAX in filters (performance hit)

Performance Optimization

🗄️Model Optimization

Remove unused columns: Every column uses memory. Remove what you don't need.
Use integers for keys: Integer joins are faster than string joins. ProductID (int) better than ProductCode (string).
Split datetime columns: Store Date and Time separately. Date column compresses better.
Hide unused columns: Hidden columns stay in model but don't appear in field list.
-- Power Query: Remove unused columns Table.SelectColumns( Source, {"Date", "ProductID", "Revenue"} // Keep only these ) -- Or remove specific columns Table.RemoveColumns( Source, {"Column1", "Column2"} // Remove these )

📊DAX Optimization

Use variables (VAR): Calculate once, reuse multiple times. Avoid recalculation.
-- ✗ BAD: Calculates Total Revenue 3 times Gross Margin % = DIVIDE( [Total Revenue] - [Total Cost], [Total Revenue], 0 ) -- ✓ GOOD: Calculates once Gross Margin % = VAR _Revenue = [Total Revenue] VAR _Cost = [Total Cost] VAR _Profit = _Revenue - _Cost RETURN DIVIDE(_Profit, _Revenue, 0)
Filter before iterating: Reduce rows before SUMX, FILTER operations.
-- ✗ BAD: Iterates all rows then filters Total = SUMX( FILTER( FactSales, FactSales[Year] = 2024 ), FactSales[Revenue] ) -- ✓ GOOD: Use CALCULATE to filter first Total = CALCULATE( SUM(FactSales[Revenue]), FactSales[Year] = 2024 )

📈Visual Optimization

Limit visuals per page: Max 7-10 visuals. More = slower rendering.
Use core visuals: Custom visuals are slower. Stick to built-in when possible.
Limit data in visuals: Use Top N filters. Don't show 10,000 rows in a table.
Reduce cross-filtering: Edit interactions. Not every visual needs to filter everything.

🔄Refresh Optimization

Incremental refresh loads only new/changed data. Essential for large datasets.
When to use incremental refresh:
  • Fact tables over 1 million rows
  • Full refresh takes > 30 minutes
  • Most data is historical and unchanging
Incremental Refresh Policy: - Archive data: Older than 2 years (compressed, read-only) - Refresh data: Last 2 years (refreshed daily) - Detect changes: Use modified date column Result: - Full: 10 million rows, 2 hour refresh - Incremental: 50K new rows, 5 minute refresh 95% time savings!

Best Practices Summary

📐Modeling

  • ✓ Define grain before building fact tables
  • ✓ Use star schema (not snowflake)
  • ✓ One-to-many relationships, single direction
  • ✓ Conformed dimensions across facts
  • ✓ Integer surrogate keys for relationships
  • ✓ Mark date table for time intelligence
  • ✓ Hide foreign keys in fact tables

🧮DAX

  • ✓ Default to measures (not calculated columns)
  • ✓ Use DIVIDE for division (handles zero)
  • ✓ Use VAR for complex calculations
  • ✓ Filter with CALCULATE, not FILTER + SUMX
  • ✓ Test measures with different filter contexts
  • ✓ Name measures clearly (Total Revenue, not Revenue1)
  • ✓ Organize measures in dedicated table

🎨Report Design

  • ✓ Maximum 7-10 visuals per page
  • ✓ Executive summary page (high-level KPIs)
  • ✓ Detail pages for drill-down
  • ✓ Consistent color scheme and formatting
  • ✓ Tooltips for additional context
  • ✓ Test on mobile layout
  • ✓ Use bookmarks for navigation

🔒Security & Governance

  • ✓ Implement RLS from day one
  • ✓ Use dynamic RLS (not static)
  • ✓ Test RLS with "View as role"
  • ✓ Document workspace access roles
  • ✓ Endorse datasets (promote/certify)
  • ✓ Schedule refresh during off-hours
  • ✓ Monitor refresh failures

📚Documentation

  • ✓ Data dictionary (table/column definitions)
  • ✓ Relationship diagram
  • ✓ Measure library with formulas
  • ✓ Refresh schedule and dependencies
  • ✓ RLS role assignments
  • ✓ Data source connections
  • ✓ Change log for major updates

⚡Performance

  • ✓ Remove unused columns in Power Query
  • ✓ Use integers for key columns
  • ✓ Implement incremental refresh (large tables)
  • ✓ Limit visuals per page (7-10 max)
  • ✓ Use variables in complex DAX
